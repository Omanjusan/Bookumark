## コントロール群のレイアウト

- Raycast型: 検索が主役(最上段・大きめ・オートフォーカス想定)、操作系は下部に集約
- タブ/属性フィルタは検索直下のチップ列。フッターは「表示の操作」に専念させる
- 表示形式(5種)の切り替えはセグメントボタンで常時展開(看板機能なので隠さない)
- ソートUIは2案をモックで比較中、未決:
  - 案A: 軸ドロップダウン + 昇順/降順トグルボタン
  - 案B: 4軸チップ横並び、選択中チップの再クリックで昇降反転
- 自由移動トグルON → ソートは「カスタム」に固定され、ソートUIは無効化(バッジ表示)
- テーマはライト/ダーク両対応。CSS Custom Propertiesでトークン化し、テーマ差し替え設計の基盤にする

## ドッキングベイシステム

コントロール群は「ベイ」単位で分割し、⠿ つまみのドラッグでレール間を自由に再配置できる。

- ベイは5つ: 検索 / フィルタ / 表示形式 / ソート / ツール(自由移動・テーマ・設定)
- レールは横4本: アイテム欄を挟んで上2本+下2本
- 空レールは普段畳まれ、ドラッグ中のみ点線で出現。挿入位置はインジケータ表示
- 配置は保存して起動時に復元(本実装では `browser.storage.local` にレール→ベイ順の JSON)
- 実装方針: 各ベイを custom element 化。HTML Drag & Drop API(または Pointer Events)で実現可能なことはモックで確認済み

### 後回し(将来対応)

- **左右の縦レール(サイドドック)**: 上下だけでなく左右にもドッキング反応エリアを置き、縦配置を可能にする。
  ただし縦配置にするとベイはアイコン主体のUIに変える必要があるため、横レールの完成後に着手する
- つまみの表示ポリシー(常時 / ホバー時 / レイアウト編集モード時のみ)は未決
- キーボードのみでベイを移動できる代替手段(アクセシビリティ)を本実装で用意する

## パネル式スケール表示(v0.3〜v0.4で策定)

- サイズは5段階: **1/4, 1/2, 1(等倍=公式スピードダイヤルアイコン), 2, 4**
  - 最大=縦横2倍(4倍面積)、最小=縦横1/2(1/4面積)。中間の1/2と2は 2:1 の長方形
  - グリッドセル単位: 1/4=1×1, 1/2=2×1, 1=2×2, 2=4×2, 4=4×4 → 2倍差の組み合わせで隙間なく敷き詰め(CSS Grid dense flow)
- スケールが効くのはスケーラブル軸(現状は訪問回数のみ)選択時。非スケーラブル軸では全タイル等倍
- タイルサイズごとの情報量: 1/4=アイコンのみ / 1/2=+タイトル / 2以上=+訪問回数

### スロット定員制(カスケード降格)

最大サイズを無制限に許すとUIが大パネルに占有されUXが崩れるため:

- 訪問回数のしきい値は「希望サイズ」を決めるだけ。サイズ段階ごとに**スロット定員**を持つ
- 訪問回数の多い順に割り当て、満員なら**次のサイズへ強制降格**(近似値でも容赦なくカスケード)
- 定員は**画面サイズ(グリッド列数 n)から動的算出**。仮の係数: 4倍=n/6, 2倍=n/3, 1倍=n, 1/2=2n(各下限あり)、1/4=無制限 → 要チューニング
- リサイズ時は定員を再計算してレイアウトし直す

## 表示形式の実装メモ

### アイコン式(v0.5・確定)

- 等倍52pxアイコン(角丸14px)+ラベル最大2行(11px)を余白多めのグリッドに整列(Notion的)
- グリッド: `repeat(auto-fill, minmax(86px, 1fr))`、行間広め(gap 20px 10px)
- スケールは効かない。どのソート軸でも均一サイズ(スケールはパネル式専用)
- 訪問回数は表に出さずツールチップのみ。ホバーで淡い背景ハイライト

### カード式(v0.6・大きめ)

- 1枚のカード = 48pxアイコン + タイトル(13.5px 太字・1行省略)+ ドメイン(10.5px)+ メタ行(訪問回数・最終アクセス)
- グリッド: `repeat(auto-fill, minmax(270px, 1fr))` の可変列、カード min-height 96px。surface 背景+ボーダー、ホバーでボーダー強調+浮き
- **ユーザーメモの余地を確保**: カード下部に2行クランプのメモ領域(`.card-memo`)を予約済み。データがあるときだけ表示される構造で、メモ機能自体は未実装(将来対応)
- 等倍のみ(スケールなし)。利用実績データをテキストで常時見せる最初の形式=アイコン系とテキスト系の中間
- 実装ではドメイン表示用に URL のホスト名が必要(モックでは擬似ドメインマップで代用)

## manifest / 権限(最小E2E版・決め打ち)

manifest.json はリポジトリ直下。web-ext lint でエラー0を確認済み(2026-07-20)。

- **Manifest V3**、`browser_specific_settings.gecko.id = bookumark@omanjusan`、`strict_min_version = 115.0`
- `data_collection_permissions.required = ["none"]` — データ収集なしを明示(新しいAMO要件)。
  lint警告2件は Firefox for Android 対応バージョン絡みのみ → デスクトップ専用なので無視でよい
- **permissions は3つで確定**:
  - `bookmarks` — コア層の読み取り+削除の書き戻し(正本=公式DB)
  - `history` — 訪問回数・最終訪問日時の読み取り(常に読み取り専用)
  - `storage` — オーバーレイ層(並び順・属性・メモ・表示設定)の保存
- 不要と判断したもの:
  - `tabs` — ブックマークを開くだけなら `tabs.create()` に権限不要
  - `host_permissions` — 外部通信なし
  - favicon 取得 — E2E版はグリフ(頭文字バッジ)で代替。取得手段は後日検討
- **起動形態(E2E版)**: `npm run experiment`(= `web-ext run`)で Firefox が立ち上がり、
  background の `runtime.onInstalled` が `panel/panel.html` をタブに自動で開く超シンプル構成。
  ツールバーボタン(`action.default_popup`)からも同じ panel.html が開ける。
  要求定義の「スピードダイヤル形式で起動」(`chrome_url_overrides.newtab`)は侵襲性が高いため後続で選択式にするか検討
- **background は開発用の1リスナーのみ**(インストール時にパネルのタブを開くだけ)。
  孤児オーバーレイの即時削除は、E2E版ではパネル起動時の突き合わせ(lazy reconcile)で担保。
  `bookmarks.onRemoved` 等のイベント購読は後続で導入
- 開発ツール: `web-ext` を devDependency 化(`npm run experiment` / `npm run lint:ext`)。lint エラー0・警告0

## TypeScript化(Phase 5・完了)

- バンドラ無し。`src/` に `.ts` ソースを置き、`tsc`(`rootDir: src`, `outDir: .`)で
  既存の実行時パスへそのまま出力: `src/background.ts`→`background.js`、
  `src/panel/panel.ts`→`panel/panel.js`、`src/panel/lib/*.ts`→`panel/lib/*.js`
- import は `./lib/bookmarks.js` のように **`.js` 拡張子で書く**(`moduleResolution: "Bundler"`)。
  ブラウザがネイティブESMとして相対URL解決するため、出力側の拡張子がそのまま合っている必要がある
- 型定義は `@types/firefox-webext-browser` を devDependency に追加。`browser.*` がグローバルに型付けされる
- `tsconfig.json` の注意点: `outDir` がプロジェクトルート(`.`)で `rootDir`(`src`)を包含するため、
  デフォルトの自動除外ルールが `src` ごと除外してしまう → `exclude: ["node_modules"]` を明示して回避
- `npm run build`(= `tsc`)を追加。`experiment` は変更なし(CLAUDE.md定義の `web-ext run` のまま)。
  確認は `npm run build && npm run experiment`
- `*.js.map` は生成物のため `.gitignore` に追加。コンパイル後の `.js` 自体(background.js, panel/**/*.js)は
  従来どおりリポジトリ直下に存在する実行時アセット(コミット可否はユーザー判断)

## 実装候補(次ステップ)

**最初に作るのは最小E2E版**(開く/並べる/削除のみ・ビュー1種・ドック無し)。スコープと正本方針は [migration.md](migration.md) の決定事項を参照。

モックで最低限の仕様は確定。以下を実装・検討候補として持ち越す:

1. スロット定員の係数チューニング(4倍=n/6, 2倍=n/3, 1倍=n, 1/2=2n は仮置き。パネル幅スライダーで触りながら調整)
2. タイル内情報量の段階設計の詰め(現状: 1/4=アイコンのみ / 1/2=+タイトル / 2以上=+訪問回数)
3. 残り2形式のデザイン策定(タイトル式 → 詳細式。アイコン式・カード式はモック済み→「表示形式の実装メモ」参照)
4. 最終アクセス日時をスケーラブル軸に含めるかの判断
5. ソートUIの最終決定(案A: ドロップダウン / 案B: チップ)
6. つまみの表示ポリシー(常時 / ホバー時 / 編集モード時のみ)
7. 左右の縦レール(サイドドック)— アイコン主体UIが必要なため横レール完成後
