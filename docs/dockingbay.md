# ドッキングベイ・チップカスタマイズ詳細設計

策定日: 2026-07-22

## 1. 目的

Bookumarkの検索、フィルタ、表示形式、ソートおよびツール類を、再配置可能な「ベイ」と、ベイへ配置する「チップ」に分離する。

本書では、固定ベイによる機能検証から、ベイ工場、名前付きレイアウト、ドッキングD&D、保存・復旧までの仕様を定義する。メインUIは通常操作に専念させ、チップ編集用UIをモーダルへ隔離する。

本書は、[features.md](features.md) の「ドッキングベイシステム」を具体化した確定仕様である。同文書にある「上下4本、左右レールは後続」という旧案に対し、本書では **上・左・右・下の4レール** を最終構成とする。

## 2. 基本方針

- 最初にD&Dなしの固定ベイへ既存・予定UIを並べ、各機能の振る舞いを確定してテストする。
- 次にチップをデータ駆動のコンポーネントへ移行し、ベイ工場で編集可能にする。
- 最後に名前付きレイアウト、ベイの追加・再配置・未配置化およびD&Dを導入する。
- ベイ内部の編集と、メインUI上のベイ配置は別の設定・保存単位とする。
- メインUI上の各ベイへ編集ボタンを常設しない。
- チップおよびベイは同種の重複を許容し、競合を事前に拒否するガードレールを設けない。
- 初期版の編集操作はマウスD&Dへ依存してよい。キーボード代替操作は後続とする。

## 3. 用語

| 用語 | 定義 |
| --- | --- |
| チップ種別 | 検索、フィルタ、ソートなど、チップが提供する機能の種類 |
| チップインスタンス | ベイへ配置された個別のチップ。同じ種別を複数作成できる |
| ベイ | 0個以上のチップを順序付きで保持するUIコンテナ |
| デフォルトベイ | 基本チップを固定搭載する、編集・削除不能な保険用ベイ |
| ベイ工場 | 1つのユーザーベイを編集するモーダルダイアログ |
| レール | メイン表示領域の上、左、右、下にあるベイ配置領域 |
| 名前付きレイアウト | ベイの配置、向き、表示状態を用途別に保存する設定 |
| 内部デフォルトレイアウト | 初期状態および復旧に使う編集・削除不能なレイアウト |
| 未配置 | ベイ定義は存在するが、現在のレイアウトのどのレールにも属さない状態 |
| 条件チップ | 配置順に設定値を状態へ合成するチップ |
| 操作チップ | 共通アプリ状態を表示し、ユーザー操作時に更新するチップ |
| アクションチップ | クリックなどの明示操作時だけ処理を実行するチップ |

## 4. スコープ

### 4.1 対象

- D&Dなしの固定ベイ
- チップの共通データ契約と3分類
- 同種チップの複数配置
- ベイ工場モーダル
- チップツールセレクター
- チップの追加、削除、並べ替え、設定変更
- ユーザーベイの作成、改名、複製、完全削除
- デフォルトベイ
- 上、左、右、下の4レール
- 名前付きレイアウト
- レイアウトの作成、改名、複製、削除、切り替え、既定指定
- ベイピッカー
- ベイの配置、再配置、未配置化
- ベイ内部設定とメインレイアウトの分離保存
- Undo、Redo、明示保存、破損時フォールバック
- 廃止チップの検出と削除

### 4.2 対象外・後続

- タッチ端末対応
- キーボードだけによるチップ・ベイのD&D代替操作
- 廃止チップから後継チップへの自動データ移行
- ユーザー定義チップのスクリプト実行
- ベイやチップ設定の外部インポート・エクスポート
- クラウド同期
- レイアウトの自動折り畳み、自動別レール移動
- チップ競合を検出・禁止するガードレール

## 5. データ契約

### 5.1 ID

- ベイ、チップインスタンス、名前付きレイアウトは表示名と別に変更不能な内部IDを持つ。
- ベイIDは `bay-1`, `bay-2` のような単調増加値とする。
- 削除済みIDを再利用しない。
- 表示名は変更可能とする。
- ベイ名は重複可能、レイアウト名は重複時に連番を自動付与する。

### 5.2 型の概念

```ts
type ChipKind = "condition" | "control" | "action";
type RailId = "top" | "left" | "right" | "bottom";

type ChipInstanceConfiguration = {
  instanceId: string;
  chipType: string;
  kind: ChipKind;
  order: number;
  settings: Record<string, unknown>;
};

type BayConfiguration = {
  id: string;
  name: string;
  permanent: boolean;
  chips: ChipInstanceConfiguration[];
};

type BayPlacement = {
  bayId: string;
  rail: RailId;
  order: number;
};

type LayoutConfiguration = {
  id: string;
  name: string;
  systemDefault: boolean;
  placements: BayPlacement[];
};

type DockingState = {
  schemaVersion: number;
  nextBaySequence: number;
  bays: BayConfiguration[];
  layouts: LayoutConfiguration[];
  activeLayoutId: string;
  preferredLayoutId?: string;
  lastUsedLayoutId?: string;
};
```

実装時はベイ内部設定とメインレイアウトを論理的に別キーへ保存する。

```text
bayConfigurations.v1
mainLayouts.v1
dockingMetadata.v1
```

複数キーを同時に変更する新規ベイ作成などは、可能な限り1回の `browser.storage.local.set()` で確定する。

## 6. チップの状態モデル

### 6.1 複数配置

- 同じ `chipType` を持つチップを何個でも配置できる。
- 各インスタンスは別の `instanceId` を持つ。
- 同種の重複判定や排他制御は行わない。
- 競合結果は配置順によって決まり、ユーザー責任とする。

### 6.2 条件チップの適用順

ベイの適用順は次とする。

```text
上レール → 左レール → 右レール → 下レール
```

各レール内では配置順にベイを処理する。横ベイ内は左から右、縦ベイ内は上から下へチップを処理する。

```ts
for (const bay of orderedVisibleBays) {
  for (const chip of bay.orderedConditionChips) {
    state = chip.apply(state);
  }
}
```

- 同じ状態キーへ複数チップが値を設定した場合、後から適用した値を採用する。
- 異なる状態キーは両方を保持し、合成する。
- 同順は存在させない。
- 未配置ベイおよび削除されたチップは適用対象外とする。

### 6.3 操作チップ

操作チップはインスタンス自身に選択値の正本を持たず、チップ種別に対応する共通アプリ状態を参照・更新する。

```text
複数のレイアウト選択チップ → activeLayoutId
複数の表示形式チップ       → activeViewType
複数のソート方向チップ     → activeSortDirection
```

レイアウト選択チップは読み込み時に別レイアウトを自動適用しない。現在の `activeLayoutId` を表示し、ユーザー操作時だけ切り替える。これによりレイアウトAとBが互いを選択する循環を防ぐ。

### 6.4 アクションチップ

アクションチップは初期化時の状態合成へ参加せず、クリックなどの明示操作時だけ処理する。

### 6.5 チップの削除

- チップの「非表示」状態は設けない。
- ベイ工場でチップをベイ外の空き領域へドラッグしてドロップすると削除する。
- チップツールセレクターからベイへドラッグすると、新しいインスタンスを生成する。
- 追加・削除・並べ替え・設定変更はUndo・Redo対象とする。

## 7. デフォルトベイ

- `permanent: true` を持つデフォルトベイを1つ用意する。
- ID文字列ではなく `permanent` 属性で特別扱いを判定する。
- 基本チップを固定搭載する。
- チップの追加、削除、並べ替え、設定変更を禁止する。
- ベイ工場の編集対象ドロップダウンへ表示しない。
- ベイ定義の削除を禁止する。
- 名前付きレイアウトから未配置にすることは許可する。
- 内部デフォルトレイアウトでは常に復旧入口として配置する。

## 8. ベイ工場モーダル

### 8.1 開き方

メインUIのグローバルな「ベイを編集」から編集対象選択モードへ入る。

- ドロップダウンで対象ベイを選ぶと、メインUI上の該当ベイをアウトライン表示する。
- 画面上のベイをクリックしても選択でき、ドロップダウンと双方向同期する。
- 選択だけではモーダルを開かない。
- 「選択したベイを編集」、Enterまたはダブルクリックでベイ工場を開く。
- デフォルトベイは選択候補から除外する。

### 8.2 1対1編集

- 1回のモーダルで1つのユーザーベイだけを編集する。
- 工場内に複数ベイを配置するキャンバスは持たない。
- メインUI上の各ベイへ編集ボタンを付けない。
- 閉じる操作は右上の `×` に集約し、下部に閉じる・キャンセルボタンを置かない。
- Escapeは `×` と同じ閉じる処理を通す。

### 8.3 メニューバー

上部へ次のメニューバーを常設する。

```text
[ベイ名] [元に戻す] [やり直す] [保存]                         [削除]
```

- 保存は状態を確定するだけで、モーダルを閉じない。
- 未保存状態で `×` を押した場合だけ「編集を続ける／変更を破棄」を表示する。
- 削除はベイを削除予定状態にする。保存時に完全削除を確定する。
- 削除予定状態は保存前ならUndoできる。
- 削除保存後は「ベイを削除しました」という完了表示へ切り替え、ユーザーが `×` で閉じる。
- 複数レイアウトから参照されるベイでは、削除されるレイアウト数を表示する。

### 8.4 ツールセレクター

- Photoshop型の縦ツールセレクターをモーダル左側へ常設する。
- アイコンをカテゴリ別に区切って縦に並べ、必要ならスクロールする。
- ホバー開始150ms後に名前と説明のツールチップを表示する。
- ドラッグ開始時にチップ実体プレビューを生成する。
- ベイへドロップすると新規チップインスタンスを作る。
- 同じツールを何度でもドラッグして複製できる。

### 8.5 縦置きの編集

- メインUIで縦置きされるベイも、工場では横ベイとして編集する。
- 工場の左側が実表示の上、右側が実表示の下へ対応する。
- 空ベイには「左側が上部、右側が下部に回転表示されます」相当の案内を表示する。
- 文字とアイコンは縦配置時も正立させる。
- 通常幅がベイ幅を押し広げるチップは縦専用短縮表示へ変換する。
- 短縮表示で不足する操作は、メイン表示領域側への横方向エクスパンションで提供する。

## 9. ベイの作成・複製・完全削除

### 9.1 新規作成

```text
「ベイを追加」
  → 新規ベイ工場を開く
  → チップを編集
  → 保存成功
  → 新IDを確定
  → 現在レイアウトへ初期配置
```

保存またはキャンセル前の空ベイをメインUIへ仮配置しない。

初期配置は次の順で空きを探す。

```text
上レール末尾 → 下レール末尾 → 左レール末尾 → 右レール末尾
```

全レールに収まらなければ保存は成功させ、現在レイアウトでは未配置にする。

### 9.2 複製

- 複製はユーザーの明示操作とし、削除時の自動複製はしない。
- 元ベイのチップ構成と設定をディープコピーする。
- 新しいベイIDを発行する。
- 名前は「元の名前 2」のように自動採番する。
- 複製後は元ベイと独立して編集できる。
- 複製操作はUndo・Redo対象とする。

### 9.3 完全削除

- ベイ工場の削除メニューだけが完全削除を行う。
- ベイ定義、所属チップ、値、全レイアウトの参照を削除する。
- メインUIのレイアウト編集で中央ゴミ箱へ落とす操作は完全削除ではない。
- `permanent: true` のベイは完全削除できない。

## 10. メインレイアウト

### 10.1 レール

| レール | ベイ方向 | スクロール |
| --- | --- | --- |
| 上 | 横 | 横方向 |
| 左 | 縦 | 縦方向 |
| 右 | 縦 | 縦方向 |
| 下 | 横 | 横方向 |

- レールは一次元の配置順だけを持つ。
- ベイの重なりやZ値を扱わない。
- リサイズ時に別レールへ自動移動させない。
- 自動折り畳みを行わない。
- 収まらない場合は、先にベイ間やレール内の空白を削減する。
- それでも収まらなければレール内スクロールを使う。
- スクロール可能なレールでは、端ドラッグによるパン・自動スクロールを提供する。

### 10.2 レイアウト編集モード

- 通常のチップ操作とブックマークを開く操作を無効化する。
- ベイの選択、配置、移動、未配置化だけを受け付ける。
- メニューバーは次とする。

```text
[レイアウト名] [元に戻す] [やり直す] [保存]                  [削除]
```

- 編集内容はドラフトとして保持し、保存まで確定しない。
- 未保存状態で別レイアウトへ切り替える場合は「編集を続ける／変更を破棄」を表示する。
- 保存後を不可逆のUndo・Redo開始点とする。
- 保存操作自体は履歴へ加えない。
- 内部デフォルトレイアウトでは編集ボタンを排除し、「デフォルトレイアウトでは保存・削除・変更が禁止されています」と表示する。

### 10.3 中央ゴミ箱

- ベイのドラッグ中だけ、メイン表示領域中央にゴミ箱エリアを表示する。
- ドロップ可能時は明確にハイライトする。
- ドロップすると、現在レイアウトからベイを未配置にする。
- ベイ定義とチップ設定は残す。
- メインUI上のベイに `×` は付けない。

### 10.4 ベイピッカー

ベイを縮小描画せず、ドラッグ可能なネームタグとして一覧する。

```text
未配置
[ベイA] [ベイC]

----------------

配置済み
[ベイB] [ベイD]
```

- 上段は現在レイアウトの未配置ベイ、下段は配置済みベイとする。
- ドラッグ開始時にベイ実体プレビューを生成する。
- 配置候補レールに応じて横・縦プレビューを切り替える。
- 配置済みタグをドラッグ中は、元のベイを半透明にする。
- 再配置成功時は元位置から移動し、新位置へ表示する。
- Escape、ドロップ中止または中央ゴミ箱へのドロップでは未配置または元配置へ戻す。
- クリック選択だけで配置する場合は、上、下、左、右の空き順を使う。
- 同じベイIDは1レイアウト内へ1回だけ配置できる。

## 11. 名前付きレイアウト

### 11.1 管理機能

- 新規作成
- 名前変更
- 複製
- 削除
- 選択・切り替え
- 設定画面からの既定指定
- レイアウト選択チップからの切り替え

レイアウト名が重複する場合は連番を自動付与する。

### 11.2 新規作成と複製元

- 新規レイアウトの初期選択は空白とする。
- 作成時に複製元をドロップダウンから選べる。
- 複製元候補には内部デフォルトレイアウトも表示する。
- 複製元を選んだ場合、次の2択を提示する。

```text
● 同じベイを使用
  ベイの編集内容が両方のレイアウトに反映されます

○ ベイも複製
  新しいレイアウト専用のベイとして編集できます
```

- 「同じベイを使用」は同一ベイIDへのリンク配置とする。
- 「ベイも複製」はユーザーベイへ新IDを発行し、内容をディープコピーする。
- デフォルトベイは複製せず、常に同じ `permanent` ベイを参照する。
- 既定値は「同じベイを使用」とする。

### 11.3 内部デフォルトレイアウト

- 初期状態とフォールバックに使う。
- 削除・改名・保存・配置変更を禁止する。
- 通常のレイアウト選択ドロップダウンへ常駐させない。
- 「デフォルトに戻す」専用ボタンで一時的に切り替える。
- 名前付きレイアウトを上書きしない。
- 新規レイアウトの複製元選択時だけ候補へ表示する。

### 11.4 起動・削除時のフォールバック

起動時は次の順に選ぶ。

```text
最後に使用した有効なレイアウト
  → 設定画面で指定した既定レイアウト
  → 内部デフォルトレイアウト
```

- 現在のレイアウトを削除した場合も同じ順で切り替え、メインUIを再構築する。
- 既定指定されたレイアウトを削除した場合は既定指定も解除する。
- 名前付きレイアウトをすべて削除しても内部デフォルトで稼働する。

## 12. 保存、再構築、Undo・Redo

### 12.1 保存単位

- ベイ工場は対象ベイのチップ構成と設定を保存する。
- レイアウト編集はベイのレール、順序、未配置状態を保存する。
- 両者は別々に保存し、片方の変更が他方を暗黙に書き換えない。

### 12.2 保存成功

- 保存はモーダルや編集画面を閉じない。
- 保存時点を新しい不可逆のUndo・Redo開始点とする。
- 保存以前へのUndoを許可しない。
- 保存操作自体を履歴へ記録しない。
- メインUIの検索、フィルタ、ソートなどの実行中状態を破棄する。
- 保存データを再読込し、ベイとチップを適用順に再評価してメインUIを再構築する。

### 12.3 保存失敗

- 編集画面を閉じない。
- ドラフトとUndo・Redo履歴を保持する。
- メインUIを再構築しない。
- [todo.md](todo.md) の統合通知機能を通してエラーを表示する。

## 13. 廃止・破損・実行時エラー

### 13.1 廃止台帳

チップ実装を削除した後も、廃止台帳を永続的に保持する。

```ts
type DeprecatedChipRecord = {
  chipType: string;
  displayName: string;
  deprecatedSince: string;
  removedSince: string;
  replacementChipType?: string;
};
```

- 現在の保存データと台帳を照合するため、飛び級アップデートでも検出できる。
- 前回起動以後に削除対象となった全チップを、ベイ名と件数付きで1つのダイアログへ列挙する。
- 対象チップは通知中も描画・機能適用しない。
- 「確認」プッシュボタンを置き、押すと「確認した」状態へ変える。
- 確認されるまで起動時に通知し続ける。
- 確認後、対象の全インスタンスを削除する。
- 廃止台帳自体は削除しない。
- `replacementChipType` があれば代替候補を案内する。
- 設定構造の変換がチップごとに異なるため、自動置換は後続フェーズとする。

### 13.2 未知のチップ・破損データ

- 現行一覧にも廃止台帳にもない `chipType` は破損データとして扱う。
- 「該当箇所にエラーを検知しました。稼働可能な状態までフォールバックを行います」相当のダイアログを表示する。
- 正常なベイ・レイアウト・チップを維持し、破損部分だけを除外する。
- 部分正常化後の状態を保存し、同じエラーを繰り返さない。
- 正常化できなければ内部デフォルトレイアウトへフォールバックする。

### 13.3 実行時エラー

- 1つのチップでエラーが発生しても画面全体を停止しない。
- 対象チップだけをその評価で無効化し、残りのチップを順番どおり適用する。
- 統合通知へ対象ベイ名、チップ名、エラー概要を渡す。

## 14. TDD実装フェーズ

各Red、Green、必要時のRefactorは独立したGO、実行、テスト、報告、STAYの単位とする。固定ベイで各機能を先に成立させ、カスタマイズとD&Dを後付けする。

| フェーズ | 内容 | 主な完了条件 | コスト |

| DB-7 | ベイ編集トランザクション | ドラフト、保存、破棄、Undo・Redo、保存失敗を扱える | 大 |

## 15. テスト観点

### 15.1 チップ適用

- 同じ種別を複数配置できる。
- 同じキーは後方の値が残る。
- 異なるキーは合成される。
- 上、左、右、下の順で評価される。
- 横ベイは左から右、縦ベイは上から下へ評価される。
- 未配置ベイは評価されない。
- 1チップの例外後も残りを評価する。
- 操作チップの複数インスタンスが同じ共通状態を表示する。
- レイアウト選択チップが初期化だけで別レイアウトを選ばない。

### 15.2 ベイ工場

- 対象選択とアウトライン、ドロップダウンが同期する。
- 選択だけではモーダルを開かない。
- Enter、ダブルクリック、編集開始ボタンで開く。
- デフォルトベイを編集対象にできない。
- 同じツールを複数回ドラッグして別インスタンスを作れる。
- ベイ外ドロップでチップを削除できる。
- Undo・Redoで追加、削除、順序、設定を復元できる。
- 未保存時の `×` とEscapeで破棄確認が出る。
- 保存成功でモーダルが閉じず、履歴開始点が更新される。
- 保存失敗でドラフトが残る。
- 削除予定をUndoでき、保存で全参照が消える。

### 15.3 レイアウト

- 同じベイIDを同一レイアウトへ重複配置できない。
- 同じベイIDを異なるレイアウトから参照できる。
- リンク配置の編集が参照先すべてへ反映される。
- 独立複製が新IDを持ち、元ベイへ影響しない。
- 未配置と配置済みのタグが正しく分離される。
- 配置済みタグのドラッグ中だけ元ベイが半透明になる。
- 中央ゴミ箱で定義を残したまま未配置になる。
- 上、下、左、右の初期配置順が守られる。
- 全レール満杯なら未配置になる。
- リサイズでベイが勝手に別レールへ移らない。
- スクロール可能時だけ端パンが動く。
- 編集中は通常チップとブックマーク操作が無効になる。

### 15.4 保存・復旧

- ベイ設定の保存がレイアウト配置を変更しない。
- レイアウト保存がベイ内部設定を変更しない。
- 保存後の再構築で検索・フィルタ・ソートの実行中状態が破棄される。
- 最終使用、既定、内部デフォルトの順で復元される。
- 現在レイアウト削除時に安全なレイアウトへ切り替わる。
- 既定レイアウト削除時に既定指定も解除される。
- 一部破損時に正常部分を維持する。
- 全体を正常化できない場合に内部デフォルトが表示される。

### 15.5 廃止移行

- 1バージョン更新で廃止対象を列挙できる。
- 複数バージョンの飛び級更新でも台帳から列挙できる。
- 同種の複数インスタンスを1件の通知へ集約できる。
- 確認前は対象を描画・適用せず、次回も通知する。
- 確認後は全対象を削除し、通知を繰り返さない。
- 代替候補を表示しても自動変換しない。

### 15.6 Firefox実機

- 横レールと縦レールでチップの文字・アイコンが正立する。
- 縦専用短縮表示がベイ幅を押し広げない。
- エクスパンションがメイン表示領域側へ開き、画面端で補正される。
- ブラウザズーム時にもドロップ位置とプレビューが一致する。
- 多数のネームタグ、ベイ、チップで操作不能な停止が起きない。
- D&D中の端パンと挿入インジケータが安定する。

## 16. 完了条件

- 固定ベイ状態で各UI機能の単体・連携テストが成功する。
- チップの3分類と共通状態の正本が分離されている。
- 同種チップを複数置き、定義された後方優先で適用できる。
- ベイ工場がメインUIを編集装飾で汚染せず、1ベイを編集できる。
- ベイ内部設定と名前付きレイアウトが別々に保存される。
- 4レールへベイを配置・再配置・未配置化できる。
- デフォルトベイと内部デフォルトレイアウトから必ず復旧できる。
- リンク配置と独立複製が区別される。
- 保存失敗、破損、未知チップ、廃止チップから稼働可能な状態へ戻れる。
- Undo・Redoが保存境界を越えない。
- 単体、DOM、統合、Firefox実機テストが成功する。

## 17. 実装時の調整事項

以下はプロダクト仕様の未決事項ではなく、各TDDフェーズでfixtureと実機結果から決める実装定数である。

- レールの収容判定に使う最小余白
- D&D挿入位置の境界式
- 端パンの反応領域、開始遅延、速度
- 縦用短縮チップの具体寸法
- 横方向エクスパンションの最大幅と画面端補正
- 大量ベイ・チップに対する描画基準値
- ブラウザズームと小数ピクセルの丸め方
- Undo・Redoでまとめる連続操作の粒度
- モーダルのフォーカス管理
