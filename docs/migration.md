# migration / データ正本(Source of Truth)検討

2026-07-20 検討会。テーマ: **Firefox本来のブックマークを正とするか、本アドオンのデータを正とするか**。

## 前提: 扱うデータは3層に分かれる

| 層 | 内容 | 所有者 |
| --- | --- | --- |
| コア | URL・タイトル・フォルダ構造 | Firefox (`browser.bookmarks`) |
| 利用実績 | 訪問回数・最終訪問日時 | Firefox (`browser.history`) — 常に読み取り専用 |
| オーバーレイ | カスタム並び順・属性(タブ/フィルタ)・メモ・表示設定 | 本アドオン(Firefoxのデータモデルに存在しない) |

利用実績はそもそも browser.history からしか取れないため、正本論争の対象はコア層とオーバーレイ層。

## 案の比較

### 案A: Firefoxブックマークが正(ミラー+オーバーレイ方式)

アドオンはコア層を一切持たず、常に `browser.bookmarks` を読む。
オーバーレイ層だけを自前ストレージに持ち、ブックマークGUIDをキーに紐付ける。

- ✅ ブックマークツールバー・他デバイス(Firefox Sync)・他ツールと常に一致
- ✅ アドオンを外してもブックマークは無傷。導入時のインポート作業も不要
- ✅ 他の場所での追加・削除・リネームが自動反映(`bookmarks.onCreated/onRemoved/onChanged/onMoved`)
- ✅ 外部でブックマークが消されるとオーバーレイが宙に浮く(孤児レコード)→正DBの状態を素直に反映させる、特にガードレールは設けない。名前変更中であっても状態をキャンセルし消えるものは消える。フォールバックはひとつ前のカーソル位置、フォルダ消去された場所内にいるのであれば１つ上位のフォルダに移動する。root内のフォルダ消去であれば1つ前のアイテム(folder,itemに関わらず）にカーソルを合わせる。rootにアイテムが何もない場合はroot自体にカーソルが合う
- ✅ フォルダ構造と本アドオンの属性/タブモデルの対応付けを決める必要がある→初回起動時にrootにあるフォルダを属性化するかどうかを聞くYならアイテム内にタグとして書き込む、もちろん書き込み先はアドオン特有のデータである必要がある。→さらにユーザータグを作るかどうかの問い合わせ→Yなら新設UIに案内

## 決定事項(2026-07-20 検討会)

### 1. 正本: 案C ハイブリッドで確定

- Sync に直結する**公式ブックマークDBが正**。ここは揺るがさない
- 本アドオンの提供価値は「公式ブックマークの使いづらさゆえに適当な場所に保存されたものの**整理の手助け**」
- データフロー: **正(公式DB)から取り込み → 修正しやすいUIを提供 → 反映は公式DBへ書き戻し**

### 2. フォルダ→タブ対応: ユーザー選択式を強制し、入口を一本化

初回起動時に必ず通るフロー(自動投影はしない):

```text
ユーザー選択(どのフォルダをタブにするか)
  → 追加のユーザー定義はあるか?
      → Yes: 追加して継続
      → No : 出口(通常画面へ)
```

- このフローは初回のみ。ただし**ユーザー定義は設定画面からも同じフローを呼び出せる**(入口一本化の理由)

### 3. 孤児オーバーレイ: 即時削除・ガードレールなし

- 外部でブックマークが削除されたら、紐づくメモ・属性・並び順も**即時削除**
- 公式DBを正とする以上、状態を正しく反映するのが正義。猶予期間・ゴミ箱等の特殊なガードは設けない。消えるのはユーザー責任

### 4. 編集範囲と最小E2E版のスコープ

「E2Eが通る最小の設計」を優先。とりあえず動く・正常に閉じられる疎通を作ってから拡張する。

最小E2E版に含むもの:

- 操作は3点のみ: **開く / 並べる / 削除**(公式DBへの書き戻しは削除のみ発生)
- ビューモードは**実装コスト最小の1種のみ**(タイトル式=フラットなリスト表示を想定。他モードは選択不可)
- **ドック機構は実装しない**。ベイのパーツはアイテムパネル以外実装なし(検索・フィルタ・ソートUI等も含めて後続)
- リネーム・新規登録などの編集拡張は後続バージョンで段階導入
